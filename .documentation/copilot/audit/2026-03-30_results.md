### Security Checklist

- [x] SQL identifiers use bracket-escaping `[{name}]` interpolation pattern

- [x] **`DataSpark.Web` — all 15 `[HttpPost]` endpoints now have `[ValidateAntiForgeryToken]`** ✅ FIXED
- [ ] File uploads do not perform magic-byte (content signature) validation — extension check only (backlog item)
- [x] Sensitive paths sanitized via `DatabaseConfiguration`
- [x] No `Console.Write*` in any production code

### Detailed Security Issues

**SEC1–SEC5 (DataSpark.Web CSRF)**: `DataSpark.Web` has 15 state-changing POST endpoints across 5
controllers with no CSRF protection. Notably, `ChartController.cs:134` has `[ValidateAntiForgeryToken]`
explicitly commented out with the comment "Temporarily disabled for debugging" — a protection was
deliberately bypassed and not restored.

**Recommendation**: Add `[ValidateAntiForgeryToken]` to all `[HttpPost]` actions in `DataSpark.Web`.
For JavaScript-consumed AJAX endpoints, configure header-based antiforgery (`X-CSRF-TOKEN`).
Remove the commented-out attribute and restore protection immediately.
